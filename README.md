## Контрольные вопросы

### 1. Что такое ViewModel и зачем он нужен?
ViewModel - это класс, который хранит и управляет состоянием UI отдельно от интерфейса.

Он решает проблему потери данных при повороте экрана, так как переживает пересоздание Activity.

> [!WARNING]
> Без использования ViewModel данные состояния будут теряться при изменении конфигурации устройства.

---

### 2. В чем разница между MutableStateFlow и StateFlow?
- `MutableStateFlow` - изменяемый поток (можно менять значение)
- `StateFlow` - только для чтения

Зачем создавать `_uiState` и `uiState`:
- `_uiState` - приватный и изменяемый внутри ViewModel
- `uiState` - публичный и только для чтения

Backing property pattern - это подход, при котором внутреннее состояние скрыто, а наружу отдается только безопасная версия для чтения.

---

### 3. Почему в ViewModel нельзя хранить Context или View?
ViewModel живет дольше Activity.

Если сохранить ссылку на Activity, может возникнуть утечка памяти, так как система не сможет удалить Activity из памяти.

---

### 4. Чем StateFlow отличается от remember и rememberSaveable?
- `remember` - сохраняет состояние только при перерисовке
- `rememberSaveable` - сохраняет при повороте экрана
- `StateFlow` - хранит состояние в ViewModel и подходит для бизнес-логики

Когда использовать:
- `remember` - для простого UI состояния
- `rememberSaveable` - для простых данных UI
- `StateFlow` - для логики приложения и общего состояния

---

### 5. Что такое data class в Kotlin и зачем он используется для UI State?
data class - это класс для хранения данных.

Автоматически генерируются:
- `equals()`
- `hashCode()`
- `toString()`
- `copy()`

Функция `copy()` позволяет создавать новую копию объекта с изменением отдельных полей, что удобно для обновления состояния.

---

### 6. Как работает `collectAsState()` в Compose?
`collectAsState()` подписывает UI на StateFlow.

Когда значение меняется, Compose автоматически перерисовывает интерфейс.

UI обновляется, потому что происходит реактивное наблюдение за состоянием.

---

### 7. Зачем разделять код на пакеты data и ui?
- data - хранит модели и данные
- ui - содержит интерфейс и ViewModel

Это поддерживает принципы:
- разделение ответственности
- чистая архитектура
- удобство поддержки и тестирования

---

### 8. Что происходит с ViewModel при полном закрытии приложения (process death)?
ViewModel уничтожается.

Данные не сохраняются.

Альтернативы:
- SavedStateHandle
- Room (база данных)
- SharedPreferences / DataStore
